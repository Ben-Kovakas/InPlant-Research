# 01 — Prototype Implementation Plan
## In-Planted / Climate Resilient ATL — from hard-coded twin to data-driven retrofit engine

> **Scope:** Engineering plan to evolve the existing 608-line `src/App.jsx` Three.js/React prototype into a data-driven *suggestive green-retrofit engine* surfaced through the 3D twin + a next-steps report, structured so it generalizes beyond the one Atlanta demo building.
> **Timeline:** days (hackathon). Bias: **buildable-in-days over theoretical rigor.** Every constant below is a *defensible placeholder* with its FINAL.md §7 source cited; swap real coefficients in as they land.
> **Actor:** the commercial building owner (FINAL.md §1). Hero artifact: a per-building ranked package with payback, gallons, tons CO₂, and a stacked-incentive dollar figure, explorable on the twin.
> **Grounding docs:** `projectSummary.md`, `FINAL.md` (§2 metrics, §7 formulas — the spec), `ArchitectureCodeResearchAgents/ExecutiveSummary.md` (feasibility), `src/App.jsx`, `package.json`.

## 0. Headline decisions (read this first)

| Decision | Recommendation | Why |
|---|---|---|
| Client-side vs. backend | **Client-side first, with a single thin serverless proxy *only* for PVWatts** | Zero infra to demo; CORS + API-key hiding is the only real reason to touch a server. Everything else runs from precomputed JSON fixtures. |
| Live API calls during demo | **No — bake PVWatts results into a fixture at build time; keep live call as a togglable path** | Demos must not depend on a network round-trip on stage. Live call proves it works; fixture guarantees it runs. |
| Hero data-driven metric | **Solar via PVWatts** (FINAL.md §2 #4 / §7.1) | It's the one metric with a free, authoritative model that already exists; visibly tracks the sun slider; satisfies projectSummary's "at least one visibly data-driven intervention." |
| Engine language | **Plain ES modules (no TS)** | Days-level; no build-tool friction. Keep JSDoc typedefs for the data shapes. |
| State/render bridge | **Keep the existing imperative Three.js `useEffect`; inject an engine instance + a plain pub/sub store** | Rewriting into react-three-fiber is a multi-day yak-shave with no demo payoff. Wrap, don't rewrite. |
| New deps | `papaparse` (CSV fixtures, optional), nothing else required for Phases 0–2 | Three.js + React already present. Avoid adding a UI framework. |

**Two things the team must decide before Phase 2 (not blocking Phase 0/1):**
1. Composite-score **weights**: fixed defaults vs. user sliders. *Plan assumes sliders with sane defaults (§7.8 in FINAL).* 
2. **Single-building vs. portfolio** as the lead narrative. *Plan builds single-building first, portfolio as Phase 3.*

## 1. Target architecture

Three layers, one repo. The frontend stays where it is; we insert an **engine** between the twin and a **data layer** of JSON fixtures.

```
┌─────────────────────────────────────────────────────────────────┐
│  FRONTEND (existing Three.js/React twin)                          │
│   App.jsx  →  scene/ (3D build)  +  ui/ (panels, sliders)         │
│   subscribes to store; renders engine outputs; sun slider drives  │
│   recompute of time-dependent metrics                             │
└───────────────▲───────────────────────────────┬──────────────────┘
                │ outputs (Score[], deltas)      │ user inputs (sun t, weights, toggles)
┌───────────────┴───────────────────────────────▼──────────────────┐
│  ENGINE (pure functions, framework-free, unit-testable)           │
│   metrics/ (7 formula modules)  +  score.js (composite)           │
│   recommend.js (rank surfaces × interventions)                    │
│   coefficients.js (all placeholder constants + sources)           │
└───────────────▲───────────────────────────────┬──────────────────┘
                │ Building, CandidateSurface[]    │ pvwatts cache lookup
┌───────────────┴───────────────────────────────▼──────────────────┐
│  DATA LAYER (static JSON fixtures, versioned in repo)             │
│   buildings/lot0427.json  ·  pvwatts/<cacheKey>.json              │
│   incentives.atlanta.json  ·  climate.atlanta.json               │
│   (generated by scripts/ that hit PVWatts/eGRID offline)          │
└───────────────────────────────────────────────────────────────────┘
                          ▲ (optional, dev-only)
                ┌─────────┴──────────┐
                │ serverless proxy    │  /api/pvwatts → NREL (key in env)
                │ Vercel/Netlify fn   │  ONLY for live re-fetch
                └─────────────────────┘
```

**Why this split is right for a hackathon:** the engine is pure functions over plain objects, so it is trivially testable and demo-safe (no async in the hot path once PVWatts is cached). The data layer is just files you can hand-edit when a coefficient is wrong on stage. The serverless proxy is *optional polish* — if time runs out, you cut it and ship the cached fixture (see §7).

### 3.1 Solar — `metrics/solar.js` (HERO, §7.1)

```js
/**
 * @param {CandidateSurface} surface
 * @param {{location:{lat,lon}}} building
 * @param {PvwattsCache} cache  // map: cacheKey -> {ac_annual, ac_monthly, capacity_factor}
 * @returns {{ kwDc:number, kwhYr:number, monthlyKwh:number[], capacityFactor:number }}
 */
export function estimateSolar(surface, building, cache) {
  const kwDc = surface.usableAreaM2 * C.MODULE_POWER_DENSITY_KW_PER_M2 * C.PACKING_FACTOR;
  const key  = pvwattsKey(building.location, kwDc, surface.orientation); // lat,lon,kw,tilt,azimuth
  const pv   = cache[key];                       // pre-fetched; see scripts/fetchPvwatts.mjs
  if (!pv) return { kwDc, kwhYr: 0, monthlyKwh: [], capacityFactor: 0 };
  return { kwDc, kwhYr: pv.ac_annual, monthlyKwh: pv.ac_monthly, capacityFactor: pv.capacity_factor };
}
```

**PVWatts v8 wiring (used by `scripts/fetchPvwatts.mjs`, mirrored by `/api/pvwatts`):**
- Endpoint (note 2026 domain move): `GET https://developer.nlr.gov/api/pvwatts/v8.json` (was `developer.nrel.gov`, retired May 29 2026).
- Required params: `api_key`, `system_capacity` (kW DC = `kwDc` above), `module_type` (0=standard), `losses` (=`PVWATTS_LOSSES_PCT`), `array_type` (1=fixed roof mount), `tilt`, `azimuth`, `lat`, `lon`.
- Response: `outputs.ac_annual` (kWh/yr), `outputs.ac_monthly` (12 kWh), `outputs.solrad_annual`, `outputs.capacity_factor`.
- Get a free key at the NREL developer portal; store as `NREL_API_KEY` (never commit).

### 3.2 Stormwater — `metrics/stormwater.js` (§7.2)

```js
// Rv = 0.05 + 0.009 * I  (I = % impervious)  — GSMM Vol.2 form, FINAL §7.2
// WQv_ft3 = (P * Rv * A_ft2) / 12 ;  gal = WQv_ft3 * 7.48
export function estimateStormwater(surface, intervention) {
  const A_ft2 = surface.areaM2 * 10.7639;
  const Rv = 0.05 + 0.009 * surface.imperviousPct;
  const firstInchGal = ((C.FIRST_INCH_IN * Rv * A_ft2) / 12) * C.GAL_PER_FT3;
  // green roof retains a fraction of the first-inch event:
  const capturedGal = intervention === "greenRoof"
    ? firstInchGal * C.GREENROOF_RETENTION_FRAC : 0;
  const meetsFirstInch = capturedGal >= firstInchGal;  // Ch.74 Art.X compliance flag
  return { firstInchGal, capturedGal, meetsFirstInch };
}
```

### 3.3 Heat — `metrics/heat.js` (§7.3, empirical deltas — do NOT model from first principles)

```js
export function estimateHeat(surface, intervention, building) {
  const before = C.ROOF_TEMP_F[building.currentRoof] ?? C.ROOF_TEMP_F.darkMembrane;
  const afterKey = { greenRoof: "greenRoof", solar: "solarShaded", coolRoof: "coolRoof" }[intervention];
  const after = afterKey ? C.ROOF_TEMP_F[afterKey] : before;
  return { roofTempBeforeF: before, roofTempAfterF: after, deltaF: after - before }; // negative = cooling
}
```

### 3.4 Energy — `metrics/energy.js` (§7.4, empirical-% path; EnergyPlus deferred)

```js
export function estimateEnergy(surface, intervention, solarKwhYr) {
  // green roof: % cooling-energy reduction applied to a baseline cooling load estimate
  const baselineCoolingKwhYr = surface.areaM2 * 45; // PLACEHOLDER kWh/m²/yr cooling — SOURCE: FINAL §7.4 (refine w/ CDH+U)
  const savedFromRoof = intervention === "greenRoof"
    ? baselineCoolingKwhYr * C.GREENROOF_COOLING_REDUCTION_FRAC : 0;
  const kwhYr = savedFromRoof + (solarKwhYr || 0);     // solar offset counts as energy avoided
  return { kwhYr, usdYr: kwhYr * C.GA_POWER_USD_PER_KWH };
}
```

### 3.5 Carbon — `metrics/carbon.js` (§7.5)

```js
export function estimateCarbon(energyKwhYr, vegetatedAreaM2) {
  const operationalKg = energyKwhYr * C.GRID_EMISSION_KG_PER_KWH;        // dominant
  const sequestrationKg = (vegetatedAreaM2 || 0) * C.SEQUESTRATION_KG_C_PER_M2_YR; // minor
  const tonsYr = (operationalKg + sequestrationKg) / C.KG_PER_METRIC_TON;
  return { tonsYr, operationalKg, sequestrationKg };
}
```

### 3.6 Biodiversity — `metrics/biodiversity.js` (§7.6, honest proxy)

// No standard formula. Transparent proxy, labeled qualitative in the UI.
export function estimateBiodiversity(surface, intervention) {

}

### 3.7 Cost / ROI + incentive stack — `metrics/cost.js` (§7.7)

```js
export function estimateCostROI({ intervention, surface, kwDc, annualSavingsUsd, captured, building }) {
  let capex = 0;
  if (intervention === "solar")      capex = kwDc * 1000 * C.UNIT_COST.solar_usd_per_w; // $/W * W
  if (intervention === "greenRoof")  capex = surface.areaM2 * C.UNIT_COST.greenRoof_usd_per_m2;
  if (intervention === "lattice")    capex = surface.areaM2 * C.UNIT_COST.lattice_usd_per_m2;
  const incentives = stackIncentives({ intervention, capex, building }); // reads incentives.atlanta.json
  const netCapex = Math.max(0, capex - incentives);
  const paybackYears = annualSavingsUsd > 0 ? netCapex / annualSavingsUsd : Infinity;
  return { capex, incentives, netCapex, paybackYears };
}
```
**Incentive stack (`incentives.atlanta.json`, FINAL §5.1 / §7.7):** §179D deduction, **30% ITC** (solar, through 2032), Georgia Power rebate ($100K–350K/bldg/yr), Conservation Tax Credit (25% FMV, cap $500K), cool-roof exemption value (avoided re-roof), tree-recompense avoidance ($140/DBH-in). `stackIncentives()` applies the subset that matches the intervention and clamps to caps.

### 3.8 Composite Green Infrastructure Suitability Score — `score.js` (§7.8 + FINAL §6)

```js
// normalize each sub-metric 0–1 (min-max across all candidate scores for this building),
// then Score = Σ wᵢ · normalizedᵢ  across environmental + ROI + feasibility axes (FINAL §6).
export const DEFAULT_WEIGHTS = {
  stormwater: 0.18, heat: 0.12, energy: 0.12, carbon: 0.13, biodiversity: 0.05,
  roi: 0.25, feasibility: 0.15,            // ROI + feasibility added per FINAL §6 — what makes the owner act
};
export function composite(normalized, weights = DEFAULT_WEIGHTS) {
  return Object.entries(weights).reduce((s, [k, w]) => s + w * (normalized[k] ?? 0), 0);
}
```
Weights are surfaced as **user-adjustable sliders** (`ui/WeightSliders.jsx`, FINAL §7.8) so the owner can flip to "ROI-first" vs. "stormwater-first" and watch the ranking reorder live.

### 3.9 Recommendation + conflict resolution — `recommend.js`

For each surface, compute a Score for every `allowedInterventions` entry, then resolve the **solar-vs-green-roof-on-one-roof** conflict (projectSummary open question #2): keep the higher composite per surface for the "recommended package," but **retain the runner-up** so the UI can show the trade-off ("south roof: solar (0.74) edges green roof (0.69)"). `runEngine` returns both `ranked` (best-per-surface) and the full `surfaceScores`.

## 4. Wiring the twin (replacing the hard-coded readouts)

Today (App.jsx `update()`, lines 488–504) the readouts are: `r-elev` from sun height, `r-solar = 96 * height`, `r-shade = 18 / height`, and `r-drain` is a static `1,150 gal` string in the UI markup. Replace as follows.

### 4.1 Split metrics into time-independent vs. time-dependent
- **Time-independent** (computed once per building/weights change): `solarKwhYr`, `stormwaterGalYr`, `carbonTonsYr`, `capex`, `payback`, composite scores. These come straight from `runEngine()` and populate the **SuggestionPanel** + report.
- **Time-dependent** (recomputed every sun-slider move): the *instantaneous* readouts the twin shows live — **sun elevation**, **instantaneous solar capture %**, **crop shade m²**, **roof retention progress**. Compute these from `runEngine().timeDependent` evaluated at `t`:

```js
// engine/index.js — timeDependent factory used by the sun slider
export function evalTimeDependent(building, results, t) {
  const elevDeg = ELEV_MAX_DEG * Math.max(0, Math.sin(Math.PI * t));
  const captureFrac = Math.max(0, Math.sin(Math.PI * t)); // fraction of peak at time t
  return {
    elevDeg: Math.round(elevDeg),
    // instantaneous % of the surface's modeled peak AC at this sun angle:
    solarCapturePct: Math.round(captureFrac * results.buildingTotals.solarPeakFrac * 100),
    cropShadeM2: shadeAreaAt(building, t),     // geometric: canopy footprint × (1/sin(elev))
    retentionGal: results.buildingTotals.stormwaterGalYr, // annual figure shown as the "design capture"
  };
}
```

### 4.2 The wiring change in App.jsx (`onSlide` / `update`)
- Keep the existing sun math (`sunDir`, `ELEV_MAX`, day-cycle colours) — extract to `scene/sun.js` unchanged.
- In `update()`, **delete** the inline `96 * height` and `18 / height` lines and instead call `store.setSunT(tVal)`; the store recomputes `evalTimeDependent` and the `Readout.jsx` component renders the values. The 3D scene's `update()` keeps doing lights/colours only.
- Net: sun slider → `store.sunT` → engine `evalTimeDependent` → React readout re-render. Smooth because `evalTimeDependent` is cheap arithmetic (no PVWatts call in the loop).

### 4.3 Layer toggles → interventions
The existing legend rows already carry `data-layer="solar|drain|agri|lattice"` and toggle `groups[layer].visible` (App.jsx 520–534). Keep that, and additionally: when a layer toggles **off**, `store` drops that intervention from `buildingTotals` so the impact numbers and composite recompute (e.g., turn off Solar → carbon and ROI drop). This makes the toggles *meaningful*, not just cosmetic.

### 4.4 Ranked suggestions in-scene + side panel
- **Side panel (`SuggestionPanel.jsx`):** ordered list of `results.ranked` — "S+W roof → Solar · 51,200 kWh/yr · 19.4 t CO₂/yr · 6.8-yr payback · $33k incentives." Clicking a row highlights the matching scene group.
- **In-scene (`scene/overlays.js`):** extend the existing floating-tag mechanism (App.jsx `anchors`/`placeTags`, 537–547) to render an **impact badge** next to each intervention's `sceneAnchor` showing its top metric, and a highlight ring on the selected suggestion. Reuse the existing project-to-screen `placeTags` math.

## 6. Generalization path (port to another city/building)

The engine and frontend never hard-code Atlanta. To retarget:
1. **Swap `data/buildings/<newcity>.json`** — new Building + CandidateSurface objects (different lat/lon, areas, orientations). This is the only required change for solar.
2. **Re-run `scripts/fetchPvwatts.mjs`** — PVWatts is global; just new lat/lon. Solar works anywhere instantly.
3. **Swap `climate.<city>.json`** — local eGRID subregion emission factor (§7.5) and local utility $/kWh (§7.4).
4. **Swap `incentives.<city>.json`** — local/state incentive stack (the *only* genuinely city-specific scoring input; federal §179D/ITC carry over).
5. **Swap stormwater coefficient** — `FIRST_INCH_IN` and `Rv` form per the city's BMP manual (Atlanta=GSMM; others differ).
6. Feasibility narrative text per ExecutiveSummary's "Portability Note": FEMA NFHL / USDA soils / 811 / NFPA / NPS historic standards are **national** (reusable); zoning, tree ordinance, local historic boundaries are **re-sourced per city.**

A `city` field on Building selects the right `climate`/`incentives` fixtures. The seven formula functions are city-agnostic — only their *coefficient inputs* change.

## 7. Risks & cut-lines (what to fake vs. build if time runs short)

| Risk / time crunch | Cut-line (fake) | Keep (build) |
|---|---|---|
| PVWatts network/CORS issues on stage | Ship the **committed `data/pvwatts/*.json` cache**; never call live during the demo | The cache itself *is* real PVWatts data — still "data-driven" |
| Serverless proxy eats a day | **Cut `/api/pvwatts`** entirely; fetch happens only in the offline script | Solar still fully works from cache |
| Heat metric needs real LST raster | Use the **empirical `ROOF_TEMP_F` lookup** (FINAL §7.3 sanctions this) and label it "empirical estimate" | Honest delta numbers; same function signature for later LST upgrade |
| Energy model fidelity (EnergyPlus is days) | Use the **empirical-% cooling reduction** (FINAL §7.4 explicitly offers this) | A defensible kWh→$ link |
| Cost coefficients not yet sourced | Use the **PLACEHOLDER `UNIT_COST` ranges** with a visible "estimate — coefficients pending" tag | The ROI math + incentive stack structure (the actual "wow") |
| Biodiversity has no formula | Show the **habitat-m² proxy, labeled qualitative** (FINAL §7.6 says honesty reads as credible) | Don't overclaim — no fake species counts |
| Portfolio view (Phase 3) slips | Ship **single-building only**; portfolio is "future" in the pitch | The single-building hero artifact is the demo |
| Report export slips | Ship the **on-screen SuggestionPanel**; skip print/PDF | Numbers are still legible on the twin |
| Conflict resolution logic slips | Default to **highest-composite-per-surface**, drop the trade-off UI | Ranking still correct |

**The one thing you must not fake:** the **solar hero path** (Phase 1) — at least one metric must move *because of real data* when an input changes, or the "data-driven engine" claim collapses. Everything else can degrade to honest placeholders.
